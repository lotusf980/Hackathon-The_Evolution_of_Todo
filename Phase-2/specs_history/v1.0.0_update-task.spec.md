# Feature Specification: Update Task

**Feature Branch**: `3-update-task`
**Created**: 2026-02-26
**Status**: Draft
**Input**: User provides ID + new title and/or description (allow partial update – only fields that are provided are changed)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Update Task Title and Description (Priority: P1)

As a user, I want to update the title and/or description of an existing task, so that I can correct mistakes or add more details.

**Why this priority**: Core CRUD operation that enables task maintenance. Provides immediate value by allowing users to keep their tasks accurate and informative.

**Independent Test**: Can be fully tested by adding a task, updating its fields, and verifying the changes are reflected while ID and completion status remain unchanged.

**Acceptance Scenarios**:

1. **Given** a task with title "Buy milk" and description "From store", **When** user updates title to "Buy organic milk", **Then** the title changes and description remains "From store".
2. **Given** a task with title "Meeting" and empty description, **When** user updates description to "Team sync at 3pm", **Then** the description is updated and title remains "Meeting".
3. **Given** a task, **When** user updates both title and description, **Then** both fields are updated atomically.
4. **Given** a task, **When** user updates it, **Then** the task ID remains unchanged.
5. **Given** a task, **When** user updates it, **Then** the completion status (completed/not completed) remains unchanged.

---

### User Story 2 - Handle Invalid Update Attempts (Priority: P2)

As a user, I want clear feedback when I try to update a non-existent task or provide invalid data, so that I understand what went wrong.

**Why this priority**: Prevents user frustration and data corruption. Essential for usability but secondary to core update functionality.

**Independent Test**: Can be tested by attempting updates with invalid IDs and empty titles, verifying appropriate error messages.

**Acceptance Scenarios**:

1. **Given** an empty task list, **When** user attempts to update any task, **Then** system displays "Error: No tasks exist."
2. **Given** tasks with IDs 1, 2, 3, **When** user attempts to update ID 5, **Then** system displays "Error: Task with ID 5 not found."
3. **Given** a task, **When** user attempts to update title to empty string, **Then** system displays "Error: Task title cannot be empty." and no changes are made.
4. **Given** a task, **When** user enters non-numeric ID, **Then** system displays "Error: Please enter a valid number."

---

### User Story 3 - Partial Update Support (Priority: P3)

As a user, I want to update only the fields I specify without affecting other fields, so that I can make targeted changes efficiently.

**Why this priority**: Enhances usability by allowing focused edits. Lower priority because full updates also work, but partial updates improve user experience.

**Independent Test**: Can be tested by updating only title (description unchanged) and only description (title unchanged).

**Acceptance Scenarios**:

1. **Given** a task with title "A" and description "B", **When** user updates only title to "C", **Then** title becomes "C" and description remains "B".
2. **Given** a task with title "A" and description "B", **When** user updates only description to "D", **Then** description becomes "D" and title remains "A".

---

### Edge Cases

- What happens when user updates a task with very long text? → Accepted (no arbitrary limits)
- How does system handle whitespace-only titles? → Treated as empty, rejected with error
- What happens when user updates a deleted task? → Error message "Task not found"
- Does update affect task ordering? → No (tasks maintain original order/ID)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST prompt user to enter the ID of the task to update.
- **FR-002**: System MUST validate that the entered ID is a positive integer.
- **FR-003**: System MUST verify that a task with the specified ID exists before attempting update.
- **FR-004**: System MUST prompt user to enter a new title (optional - leave blank to keep current).
- **FR-005**: System MUST prompt user to enter a new description (optional - leave blank to keep current).
- **FR-006**: System MUST trim leading and trailing whitespace from new title and description before storage.
- **FR-007**: System MUST reject updates where the new title is empty after trimming and display "Error: Task title cannot be empty."
- **FR-008**: System MUST preserve the current title if user provides an empty new title (partial update).
- **FR-009**: System MUST preserve the current description if user provides an empty new description (partial update).
- **FR-010**: System MUST NOT modify the task ID during update.
- **FR-011**: System MUST NOT modify the task completion status during update.
- **FR-012**: System MUST display a success confirmation message after successful update.
- **FR-013**: System MUST display "Error: No tasks exist." when user attempts to update from an empty list.
- **FR-014**: System MUST display "Error: Task with ID {id} not found." when the ID does not match any task.
- **FR-015**: System MUST return to the main menu after update or error.

### Key Entities

- **Task**: (See Add Task spec) - No structural changes
  - Update operation modifies `title` and/or `description` fields
  - Fields `id` and `completed` are immutable during update

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can update a task's title, description, or both in under 30 seconds.
- **SC-002**: 100% of update operations on valid IDs successfully modify the correct fields.
- **SC-003**: 100% of partial updates (title-only or description-only) preserve unchanged fields.
- **SC-004**: 100% of invalid update attempts produce appropriate error messages without crashes.
- **SC-005**: Task ID and completion status remain unchanged in 100% of update operations.
