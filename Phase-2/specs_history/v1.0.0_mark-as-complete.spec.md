# Feature Specification: Mark as Complete

**Feature Branch**: `5-mark-as-complete`
**Created**: 2026-02-26
**Status**: Draft
**Input**: Toggle completion status of a task by ID (single command that flips state – no separate done/undone commands)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Toggle Task Completion (Priority: P1)

As a user, I want to toggle the completion status of a task by its ID, so that I can mark tasks as done or undo completed tasks.

**Why this priority**: Core feature for task lifecycle management. Provides immediate value by allowing users to track progress. The toggle approach simplifies the interface (one command vs. two).

**Independent Test**: Can be fully tested by adding a task, toggling its status multiple times, and verifying the completion state flips correctly each time.

**Acceptance Scenarios**:

1. **Given** an incomplete task with ID 1, **When** user toggles completion for ID 1, **Then** the task is marked as completed (`[x]`).
2. **Given** a completed task with ID 1, **When** user toggles completion for ID 1, **Then** the task is marked as incomplete (`[ ]`).
3. **Given** a task, **When** user toggles its completion status multiple times, **Then** the status alternates between completed and incomplete correctly.
4. **Given** a task is toggled to completed, **When** user views the task list, **Then** the task displays with `[x]` marker.

---

### User Story 2 - Handle Invalid Toggle Attempts (Priority: P2)

As a user, I want clear feedback when I try to toggle a non-existent task, so that I understand the operation failed.

**Why this priority**: Prevents confusion when users mistype IDs. Essential for user experience but secondary to core toggle functionality.

**Independent Test**: Can be tested by attempting to toggle invalid IDs and verifying appropriate error messages without crashes.

**Acceptance Scenarios**:

1. **Given** an empty task list, **When** user attempts to toggle any ID, **Then** system displays "Error: No tasks exist."
2. **Given** tasks with IDs 1, 2, 3, **When** user attempts to toggle ID 5, **Then** system displays "Error: Task with ID 5 not found."
3. **Given** a task list, **When** user enters a non-numeric ID, **Then** system displays "Error: Please enter a valid number."
4. **Given** a task list, **When** user enters a negative ID or zero, **Then** system displays "Error: Task ID must be a positive number."

---

### Edge Cases

- What happens when user toggles a task that doesn't exist? → Error message "Task not found"
- How does system handle rapid successive toggles? → Each toggle processed sequentially, state flips each time
- What happens to task data when toggled? → Only `completed` field changes; title, description, ID remain unchanged
- Does toggle affect task ordering? → No (tasks maintain original order/ID)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST prompt user to enter the ID of the task to toggle completion status.
- **FR-002**: System MUST validate that the entered ID is a positive integer.
- **FR-003**: System MUST verify that a task with the specified ID exists before attempting toggle.
- **FR-004**: System MUST flip the `completed` boolean field: `False` → `True` or `True` → `False`.
- **FR-005**: System MUST display a success message indicating the new state (e.g., "Task marked as complete" or "Task marked as incomplete").
- **FR-006**: System MUST NOT modify the task ID, title, or description during toggle.
- **FR-007**: System MUST display "Error: No tasks exist." when user attempts to toggle from an empty list.
- **FR-008**: System MUST display "Error: Task with ID {id} not found." when the ID does not match any task.
- **FR-009**: System MUST display "Error: Please enter a valid number." for non-numeric input.
- **FR-010**: System MUST display "Error: Task ID must be a positive number." for zero or negative integers.
- **FR-011**: System MUST return to the main menu after toggle or error.

### Key Entities

- **Task**: (See Add Task spec)
  - `completed`: Boolean field that is toggled by this feature
  - All other fields (`id`, `title`, `description`) are immutable during toggle

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can toggle a task's completion status in under 10 seconds from viewing the task list.
- **SC-002**: 100% of toggle operations on valid IDs correctly flip the completion status.
- **SC-003**: 100% of toggle operations preserve all other task fields (ID, title, description).
- **SC-004**: 100% of invalid toggle attempts produce appropriate error messages without crashes.
- **SC-005**: Task list view reflects the updated completion status immediately after toggle in 100% of cases.
