# Feature Specification: View Task List

**Feature Branch**: `4-view-task-list`
**Created**: 2026-02-26
**Status**: Draft
**Input**: Displays all current tasks in a readable table-like format showing: ID, completion marker ([ ] or [x]), title, description

## User Scenarios & Testing *(mandatory)*

### User Story 1 - View All Tasks (Priority: P1)

As a user, I want to see all my tasks in a formatted list, so that I can review what I need to do.

**Why this priority**: This is the primary way users interact with their tasks. Without visibility, users cannot know what tasks exist or their current state. Essential for the application's core value.

**Independent Test**: Can be fully tested by adding multiple tasks and verifying they all appear correctly formatted in the list.

**Acceptance Scenarios**:

1. **Given** a task list with 3 tasks, **When** user views the task list, **Then** all 3 tasks are displayed with ID, completion marker, title, and description.
2. **Given** an empty task list, **When** user views the task list, **Then** system displays "No tasks found." or similar friendly message.
3. **Given** tasks with mixed completion states, **When** user views the list, **Then** completed tasks show `[x]` and incomplete tasks show `[ ]`.
4. **Given** tasks are added in sequence, **When** user views the list, **Then** tasks are displayed in the order they were added (by ID ascending).

---

### User Story 2 - Readable Formatting (Priority: P2)

As a user, I want the task list to be clearly formatted and easy to read, so that I can quickly understand my tasks at a glance.

**Why this priority**: Good formatting improves usability and reduces cognitive load. Secondary to basic functionality but critical for user satisfaction.

**Independent Test**: Can be tested by verifying consistent column alignment, proper truncation of long text, and clear visual separation between tasks.

**Acceptance Scenarios**:

1. **Given** tasks with varying title lengths, **When** displayed, **Then** titles are aligned or consistently formatted.
2. **Given** a task with a very long description (100+ characters), **When** displayed, **Then** the description is truncated with ellipsis (e.g., "Buy groceries..." ) or wrapped across multiple lines.
3. **Given** tasks with special characters in titles/descriptions, **When** displayed, **Then** all characters render correctly without breaking the table format.
4. **Given** a task with an empty description, **When** displayed, **Then** the description column shows empty space or "-" placeholder.

---

### Edge Cases

- What happens when task list is empty? → Display friendly "No tasks" message
- How does system handle very long titles? → Truncate with ellipsis at reasonable width (e.g., 40 chars)
- How does system handle very long descriptions? → Truncate with ellipsis or wrap to multiple lines
- What happens with multi-line descriptions? → Display on single line with truncation or replace newlines with spaces
- How does system handle wide console windows? → Format adapts or uses fixed column widths

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST display all tasks in the in-memory task list when user selects "View Tasks".
- **FR-002**: System MUST display each task with the following fields: ID, completion marker, title, description.
- **FR-003**: System MUST use `[ ]` to indicate incomplete tasks and `[x]` to indicate completed tasks.
- **FR-004**: System MUST display tasks in ascending ID order (order they were added).
- **FR-005**: System MUST display "No tasks found. Add a task to get started!" when the task list is empty.
- **FR-006**: System MUST truncate titles longer than 40 characters with an ellipsis (e.g., "This is a very long t...").
- **FR-007**: System MUST truncate descriptions longer than 50 characters with an ellipsis.
- **FR-008**: System MUST replace newline characters in descriptions with spaces for single-line display.
- **FR-009**: System MUST format output in a table-like structure with consistent column alignment.
- **FR-010**: System MUST display column headers (ID, Status, Title, Description) for clarity.
- **FR-011**: System MUST return to the main menu after displaying the task list.

### Key Entities

- **Task**: (See Add Task spec) - No structural changes
  - All fields are read-only during view operation
- **TaskListView**: Presentation layer component responsible for:
  - Formatting tasks for console display
  - Handling truncation and alignment
  - Managing empty state messaging

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can view the complete task list in under 5 seconds regardless of task count (for Phase 1 scale: <100 tasks).
- **SC-002**: 100% of tasks in the task list are displayed with correct ID, completion status, title, and description.
- **SC-003**: 100% of completed tasks display `[x]` marker and incomplete tasks display `[ ]` marker.
- **SC-004**: Long titles and descriptions (100+ characters) are truncated with ellipsis without breaking the display format.
- **SC-005**: Empty task list displays a friendly, actionable message 100% of the time.
